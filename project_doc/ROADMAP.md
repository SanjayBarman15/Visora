# Visora — Build-Start Roadmap (Mock UI → Real Pipeline)

**Status:** Draft
**Relationship to other docs:** IMPLEMENTATION_WORKFLOW.md laid out Phases 0–12 at a
high level before the mock UI existed. This document zooms into the *first* real
milestone now that the mock UI is done — the concrete, ordered task list to get from
"UI with fake data" to "one real video generated end to end." Everything after that
milestone continues exactly as IMPLEMENTATION_WORKFLOW.md's Phase 2 onward already
describes — this doc doesn't replace it, it's the missing detail for its Phase 0/1.

---

## Where you actually are right now

- Mock frontend: done. Zustand store shape, chat UI, scene plan card, three-panel
  workspace, narration toggle, toast wiring — all exist against fake/local data.
- Database: fully designed and can be deployed (`mova_schema_v2.sql`), but nothing
  in it is being written to by real application code yet.
- Backend: skeleton only — `main.py` exists, every agent/rag/workers package is an
  empty `__init__.py`.
- Models: dev-tier NVIDIA NIM endpoints identified and named, but no code has
  actually called them yet.

**The gap to close:** everything currently faked in the zustand store needs to
become real API calls against a real backend that actually runs the pipeline.

---

## The one milestone that matters most right now

> **Get one real, ugly, end-to-end video generated from a real prompt — no
> retrieval, no review loop, no narration, one scene only — before building
> anything else.**

This is deliberately the same as IMPLEMENTATION_WORKFLOW.md's Phase 1, but broken
into the actual ordered steps to build it, since "build Phase 1" isn't specific
enough to start from.

## Step-by-step build order

### 1. Stand up the real backend skeleton
- Deploy `mova_schema_v2.sql` to an actual Supabase project (dev environment)
- Build out `apps/api` beyond the bare `main.py`: real FastAPI app with a router
  structure (`/projects`, `/messages`, `/scenes` at minimum)
- Wire Supabase auth verification as FastAPI middleware/dependency — every route
  should know which user is calling it
- **Definition of done:** you can call `POST /projects` from a REST client (not the
  frontend yet) and see a real row appear in Supabase's table editor

### 2. Replace the frontend's fake data with real API calls, for just this one path
- Point the zustand store's project-creation action at the real `POST /projects`
  endpoint instead of a local mock mutation
- Point the project list at a real `GET /projects` call
- **Definition of done:** creating a "new video" in the actual UI creates a real
  database row, and refreshing the page shows it in the sidebar from a real fetch —
  no other feature needs to be real yet, just this one loop

### 3. Wire one real model call, end to end
- Build a small shared client wrapper around the NVIDIA NIM API (this becomes the
  foundation every agent uses later — worth getting right once)
- Make Scout's first response real: user submits a prompt → backend calls the NIM
  endpoint for Scout's model → response is saved to the database → returned to the
  frontend and rendered as a real assistant message
- Skip multi-turn elicitation sophistication entirely for now — one call, one
  response, move on
- **Definition of done:** typing a prompt into the actual UI produces a real,
  non-scripted response from an actual model call, visible in the existing chat UI

### 4. Produce one real (single) scene plan
- Add an Orion call that takes the elicitation output and returns exactly one
  scene (not a real multi-scene plan yet — that's Phase 2, not this milestone)
- Persist it to the `scenes` table
- Return it in the shape the frontend's `ScenePlanCard` already expects — this
  should require zero frontend changes if Stage 1's store shape was designed well
- **Definition of done:** the real scene plan card renders in the real UI, sourced
  from a real Orion call, with real data in the `scenes` table

### 5. Generate real code for that one scene
- Add a Forge call: takes the one scene's description, generates Manim code, no
  retrieval and no review loop yet — whatever comes back goes straight to the next
  step
- Save the generated code to `scene_checkpoints`
- **Definition of done:** real Python Manim code, generated by a real model call,
  exists in the database for a real scene

### 6. Actually render it
- Stand up the sandboxed execution environment (start with a locked-down Docker
  container running the Manim CLI — this is worth containerizing early since it's
  also your eventual production isolation boundary, not just a dev convenience)
- Wire Celery + Redis for real (this is also the first real proof that your task
  queue setup works, not just a trivial test task)
- Run the generated code, produce a real `.mp4`, upload it to Supabase Storage,
  save the URL to `scene_checkpoints.clip_url`
- **Definition of done:** a real rendered video file exists, produced by running
  real generated code in a sandboxed process, triggered through a real Celery task

### 7. Get the real video into the real UI
- Wire the workspace stage's preview player to the real `clip_url` instead of a
  placeholder
- Wire the (already-built) SSE status streaming to reflect real task state
  (queued → generating → rendering → done) instead of a scripted animation
- **Definition of done:** submitting a real prompt in the actual app, with no
  developer intervention, results in a real one-scene video playing in the real
  workspace UI

## What this milestone deliberately does NOT include

- Multi-scene plans or parallel generation (next: IMPLEMENTATION_WORKFLOW.md Phase 2)
- The review/fix loop or Guardian safety checks (Phase 3)
- Any retrieval/knowledge base (Phase 4) — Forge generates from the prompt alone
- Narration or audio (Phase 5)
- Any memory — personal or cross-scene (Phases 6, 8)
- Any of the sophistication in the full retrieval architecture, post-generation
  editing, or hardening phases

This is intentional. The goal of this specific milestone is proving the full
vertical slice works — auth → chat → plan → code → render → playback — with the
least possible complexity in each layer. Every phase after this adds real
sophistication back in, but on top of a proven foundation instead of alongside an
unproven one.

---

## Practical notes for building this as a solo builder

- **Containerize the Manim sandbox first, not last.** It's tempting to just run
  Manim directly on your dev machine to move fast, but the sandboxing requirements
  (no network, resource limits, read-only filesystem) are architecture, not
  polish — building against a real container from the start avoids a rewrite later.
- **Don't build the shared NIM client wrapper narrowly for Scout.** Every agent
  after this reuses it — spend a bit more time here than the "just make it work for
  one call" instinct suggests, since retrofitting rate limiting and retry logic
  into six separately-written call sites later is much more painful.
- **Resist adding retrieval or the review loop "while you're in there."** The whole
  point of this milestone is isolating failure — if something breaks, you want to
  know it's the render step or the model call, not a tangle of five new features
  built at once.
- **Test with a deliberately trivial prompt first** (like the "hi" example already
  in your mock screenshots) before testing with a real complex topic — you want to
  validate the pipeline shape before stressing the model's actual generation
  quality.
