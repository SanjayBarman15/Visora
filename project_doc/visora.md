Visora is an AI-powered animation platform built on a modern full-stack monorepo architecture managed with Bun workspaces and Turborepo for the JavaScript frontend and uv workspaces for the Python backend, with Turborepo providing intelligent task orchestration, parallel execution, and build caching across all workspace packages with native Vercel remote cache integration ensuring unchanged packages are never rebuilt unnecessarily, and all application code organized under a single Git repository containing apps/web for the Next.js frontend, apps/api for the FastAPI backend, and a packages layer comprising visora-schemas for shared Pydantic models, visora-agents for the LangGraph orchestration pipeline, visora-rag for the hybrid retrieval system, visora-workers for Celery task execution, visora-db for database clients, visora-tools for external tool integrations, and visora-mcp for the Model Context Protocol server layer. The frontend is built with Next.js, TypeScript, Tailwind CSS, shadcn/ui, Zustand for client state, TanStack Query for server state, and Zod for schema validation, featuring a real-time agent status panel and scene progress dashboard powered by Server-Sent Events over a persistent FastAPI streaming endpoint with color-coded model badges displaying human-readable status messages filtered into two tiers where tier one events covering phase starts, completions, errors, and retry attempts always display and tier two internal events are visible only in a developer verbose mode toggle, with Upstash Redis pub/sub as the message bus between Celery workers and the SSE handler, and an automatic polling fallback at three second intervals that activates if the SSE connection drops, reading the latest pipeline status from a Redis key updated by each worker on every agent transition, requiring no additional infrastructure beyond the existing Upstash Redis instance already used as the Celery broker and semantic cache. The backend is a FastAPI application backed by Celery workers using Upstash Redis as the broker, operating across five dedicated queues — fast for elicitation, planning, and review agents; render for Manim scene execution; audio for ElevenLabs TTS calls and ffmpeg audio mixing with a concurrency of one on the ElevenLabs worker to respect the free tier single concurrent request limit; merge for final ffmpeg audio-video composition; and voiceover for dedicated ElevenLabs calls — each with dedicated worker pools sized independently to their compute requirements, with every Celery task designed as idempotent upsert operations using deterministic task IDs derived from project ID, scene index, and attempt number to ensure safe retries without duplicate database records or storage files, exponential backoff retry configuration with a dead letter queue for tasks exhausting their retry budget, and scene-level checkpoint persistence written atomically to the database so worker crashes allow the orchestrator to resume only failed scenes without touching completed ones. All relational data including users, projects, prompt history, scene-level checkpoint state, agent run logs, prompt version records, feedback events, and project memory documents are stored in Supabase PostgreSQL with indexes covering all query-critical columns including an HNSW index on the pgvector embedding column using cosine ops with m equal to 16 and ef_construction equal to 64, a composite B-tree index on manim_version, is_candidate, and is_stale for filtered retrieval queries, B-tree indexes on animation_category and source_type, a GIN index on the metadata jsonb column, and a GIN index on document tags, while rendered scene clips, final merged videos, voiceover audio files, and background music tracks from the curated royalty-free library are stored in Supabase Storage with a retention policy deleting intermediate clips and session-scoped temporary audio files after the final video is confirmed, and the pgvector extension powers the vector similarity search layer of the RAG system entirely within the existing Supabase PostgreSQL instance with no separate vector database required.
The AI pipeline is driven entirely by NVIDIA NIM free tier APIs and ElevenLabs across eight named model roles with strict rate limiting enforced at 30 requests per minute across all NVIDIA NIM models combined, leaving a comfortable 10 RPM buffer below the 40 RPM platform limit, implemented as per-model token buckets in Upstash Redis refilling at a constant rate so no model role starves another, with per-session budget tracking storing call counts by model role, a global generation queue implemented as a Redis sorted set ordered by request timestamp showing users their queue position and estimated wait time when capacity is unavailable rather than returning hard errors, and a priority flag on Orion's internal token bucket ensuring planning calls for new projects always acquire tokens ahead of review iteration calls from ongoing projects so new sessions never feel blocked. The rate limit allocation across all models is as follows: Orion receives 10 RPM as the primary reasoning model whose review iteration depth across parallel scenes makes it the true pipeline bottleneck; Forge receives 8 RPM as the code generation and fixing model handling parallel scene workloads simultaneously; Scout receives 3 RPM as the conversational elicitation model where human typing speed makes higher allocation unnecessary; Narrator receives 3 RPM for script writing which runs once per project in parallel with scene rendering; Guardian receives 2 RPM for lightweight safety checks on inputs and outputs; Reranker receives 2 RPM for one reranking call per scene retrieval operation; and Lens-Text and Lens-Code each receive 1 RPM as batched embedding models called infrequently relative to the generation loop, with Arbiter operating entirely outside the real-time rate limit budget as a batch-only model running in scheduled background jobs.
Scout, running meta/llama-3.2-3b-instruct, is the only model in direct conversational contact with the user, handling the entire elicitation phase as a structured requirements gathering conversation that fills a typed Pydantic state object with required fields including audience level, duration target, style preference, math inclusion flag, required versus excluded concepts, and voiceover tone preference, maintaining full conversation history across all elicitation turns in LangGraph pipeline state so it never asks the same question twice, detecting URLs in user messages and routing them through the visora-tools link ingestion pipeline which dispatches GitHub links to the GitHub MCP server for actual file and README content, YouTube links to the transcript extraction tool for caption-based content, and general URLs through Playwright-based content fetching with fallback to simple HTTP fetch, with all extracted content summarized by Scout and stored as session-scoped temporary documents in the vector store with a 24-hour TTL. Orion, running nvidia/nemotron-3-ultra-550b-a55b, serves as the primary reasoning model for the Planner Agent's scene decomposition producing a typed scene plan where each scene carries an index, title, visual description, expected Manim animation types, approximate duration, inter-scene dependency references, and transition type, and for the Review Agent's structured code analysis on medium and complex scenes, with Orion's review producing a PydanticAI-validated typed issue report with category, severity, affected code block, and suggested fix per issue. Forge, running nvidia/llama-3.3-nemotron-super-49b-v1.5, is the dedicated code generation and fixing model used by the Generation Agent and Fix Agent, handling self-review on simple scenes via Scout's complexity classification to preserve Orion's budget for complex work. Narrator, running Forge's base model under a specialized script-writing system prompt with independently versioned prompts, receives the enriched prompt, the complete scene plan, all finalized generated Manim code from completed scenes so the script precisely matches the visuals, the user's voiceover tone preference, and optionally web search results via the search MCP tool for topics requiring factual verification, producing a typed NarratorScript with per-scene text segments timed to each scene's estimated duration at 130 to 150 words per minute, dispatched as a parallel Celery task simultaneously with scene rendering so audio is ready when video rendering completes. Guardian, running nvidia/llama-3.1-nemotron-safety-guard-8b-v3, performs content safety checks on both user prompt inputs before they reach the main pipeline and generated Manim code outputs before they leave the system. Reranker, running nvidia/rerank-qa-mistral-4b, reorders the top twenty vector store candidates after similarity search to a final eight by actual relevance before context package assembly. Lens-Text, running nvidia/nv-embed-v1, generates embeddings for all natural language content, while Lens-Code, running nvidia/nv-embedcode-7b-v1, generates embeddings for all code content, with both embedding workers batching documents in groups of one hundred for efficient API usage and recording the embedding model version in the embedding_model column to support future model migration re-embedding. Arbiter, running mistralai/mistral-large-3-675b-instruct-2512 under a specialized quality evaluation system prompt, operates exclusively in the nightly quality scoring job and weekly candidate promotion job outside the real-time rate limit budget.
The shared memory architecture operates across three layers to ensure all eight models work as a coherent pipeline rather than isolated stateless calls. The first layer is LangGraph pipeline state, a comprehensive PipelineState Pydantic model stored in LangGraph's state object and checkpointed to Redis on every node transition, containing the full conversation history, enriched prompt, scene plan, per-scene memory objects tracking generated code, review issues, fix attempts, and render paths, established visual constants extracted from completed scenes, narrator script, API call counts by model role, and session metadata. The second layer is cross-scene visual memory, where after each scene's code passes dry-run a lightweight AST parsing step extracts the visual constants established — color palette definitions, coordinate systems instantiated, and module-level Manim objects defined — appending them to the PipelineState's established fields so every subsequent scene's Forge generation prompt contains a visual consistency section instructing the model to maintain the color palette, coordinate systems, and object naming conventions already established by previous scenes, preventing the visual fragmentation that occurs when each scene is generated without awareness of the others. The third layer is project-level long-term memory stored in a project_memory table in Supabase PostgreSQL, capturing preferred visual style, voice preferences, animation complexity preferences, previously used color palettes, and successful animation types for returning users, loaded by Scout at the start of each elicitation session to pre-fill known preferences without asking again. Each agent receives only the slice of pipeline state it needs through dedicated context builder functions that extract and format the relevant fields into the agent's prompt, preventing token waste and context confusion from dumping the entire state into every model call.
The full user journey begins with Scout conducting a structured conversational elicitation session loading project-level memory for returning users, after which Orion's Planner Agent decomposes the enriched prompt into a typed scene plan presented to the user as an editable scene list where scenes can be renamed, reordered by dragging, deleted, or augmented before generation is approved. On approval, all scenes are dispatched as independent parallel Celery render tasks and the Narrator script generation task is dispatched simultaneously, with each scene's pipeline running: Retrieval Agent executing concurrent PageIndex tree search over the versioned Manim documentation JSON index stored in Supabase Storage and cached in Redis alongside LangChain pgvector similarity search with composite ranking combining cosine similarity at 60 percent, quality score at 25 percent, and historical success rate at 15 percent with HNSW indexing and hard version and staleness filters, followed by Reranker reordering the top twenty candidates to a final eight before both branches merge into a structured context package; Guardian safety checking the input context; Forge generating Manim code with full visual consistency context from all previously completed scenes injected into the prompt; ast.parse syntax validation; Orion or Forge reviewing the code based on complexity routing; surgical fix application guided by the typed issue report; Manim dry-run sandboxed subprocess validation with CPU limits, memory limits, no network access, read-only filesystem, and hard timeout; and Guardian safety checking the final output, with this loop repeating up to three iterations before approval or best-effort return with warning. Once all scenes pass validation they are rendered by dedicated render workers, the Narrator script is sent to ElevenLabs for per-segment audio generation with the user's chosen voice from the curated preset library, all audio segments are assembled and mixed with optional background music from the royalty-free library or user-uploaded audio with copyright acknowledgment, ffmpeg applies automatic volume ducking reducing background music to 30 percent under voiceover segments and 60 percent in gaps using sidechain compression, and the final audio-video merge produces the complete deliverable with voiceover, background music, and all scenes concatenated in dependency order. Users can adjust voiceover volume, music volume, and track selection after generation with audio-only re-processing taking two to three seconds without triggering scene regeneration, and selective scene regeneration for any individual fix triggers only that scene's pipeline loop followed by fast ffmpeg re-merge using cached clips from all unchanged scenes with inter-scene dependency tracking automatically cascading dirty flags to downstream dependent scenes.
The hybrid RAG system maintains a continuous improvement lifecycle where successfully rendered outputs receiving no follow-up fix requests are ingested into a candidate pool after near-duplicate checking at a 0.92 cosine similarity threshold, quality scores updated nightly by Arbiter using a weighted formula combining retrieval success rate at 40 percent, source type weight at 20 percent, and current score at 40 percent, candidates promoted through Arbiter's weekly pipeline at a quality score threshold of 0.65 with a minimum of five retrieval events, and documents using APIs flagged in the api_changes table on a new Manim version release automatically marked stale and downweighted. A semantic cache in Upstash Redis intercepts retrieval queries before database access using cosine similarity matching at a 0.95 threshold against stored query embeddings with a 24-hour TTL projecting 30 to 50 percent cache hit rate on mature traffic. Full observability is provided by LangSmith tracing across every agent node, tool call, retrieval branch, token usage, and iteration cycle tagged with model role and session ID, structured Celery task logging with correlation IDs linking every task to its originating session, and a prompt versioning system enabling A/B comparisons against an offline evaluation suite measuring retrieval mean reciprocal rank and first-pass render success rate per animation category. Dodo Payments is integrated for future subscription management, infrastructure configuration lives in infra/render.yaml and turbo.json at the repo root, the frontend is deployed on Vercel with automatic Bun and Turborepo remote cache detection, the backend and workers are deployed on Render with per-service root directory configuration, and the entire codebase is developed using Git, GitHub, VS Code with a multi-root workspace configuration, and Claude Code.
